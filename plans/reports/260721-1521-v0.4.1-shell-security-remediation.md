## Executive summary

This patch addresses three connected information-disclosure weaknesses in the extension security boundary: partial shell globs could expand to protected files after the static check, bare filename and symbolic-link operands were omitted from canonical-path inspection, and sensitive bash output could reach the model without the shared redactor.

The remediation adds pre-execution controls for these shell operands and a post-execution redaction backstop. It also ensures observed command text is redacted at memory and persistence boundaries while retaining exact verify matching through a command digest.

## Remediated findings

| Finding | Previous behavior | v0.4.1 control |
|---|---|---|
| Partial shell glob disclosure | Patterns such as `.en*`, `.e??`, `.??v`, and `auth.js*` were compared as literal paths, then expanded by the shell | Shell glob intersection supports `*`, `?`, character classes, and bounded brace alternatives before execution |
| Bare filename and symlink alias disclosure | Non-dot bare operands were omitted before canonical-path resolution | Non-option filesystem operands are inspected and existing aliases are resolved to their canonical repository path |
| Attached redirection gap | Redirections without whitespace could avoid the path candidate extractor | Quote-aware redirection extraction covers attached input/output forms and ignores descriptor duplication/here-string data |
| Static shell composition gap | Chained assignments, `echo`/`printf` output composition before `xargs`, and attached ripgrep short options could hide a protected operand from a literal-only pass | Variable resolution is bounded and cycle-safe; static data composition and shared ripgrep short-option parsing feed the same protected-path checks |
| Unredacted tool output | Bash stdout/stderr and result details did not pass through the shared detector | Text result blocks and JSON-like details are redacted before return; non-text media blocks remain unchanged |
| Raw evidence command retention | The in-memory ledger could retain sensitive command text and persistence depended on caller-provided redaction | Evidence boundaries redact command text internally and retain the original SHA-256 identity for exact matching |

## Enforcement order

1. Verify the active capability lock and runtime integrity.
2. Apply exec-policy rules.
3. Deny literal protected paths.
4. Deny shell globs that can target protected paths.
5. Resolve existing filesystem operands and deny canonical protected aliases.
6. Apply confirmation policy where required.
7. Before a tool result reaches the model, filter protected path metadata and redact sensitive text/details.

Protected-path denial remains higher priority than capability filesystem allowance. The result redactor is a defense-in-depth backstop and does not replace the pre-execution deny rules.

## Security boundary and residual risk

- The extension remains a policy guard, not an operating-system sandbox.
- Static shell inspection cannot prove the behavior of every program or every runtime-generated path. Workloads containing untrusted code or credentials still require process, filesystem, network, and credential isolation.
- Pattern-based sensitive-data detection reduces accidental disclosure but cannot recognize every possible secret representation.
- Shell glob detection is intentionally conservative for protected-path candidates; unsupported or excessive brace expansion is denied rather than truncated silently.
- Result redaction operates on text blocks and JSON-like details. Binary image, audio, and resource payloads are not decoded or rewritten.
